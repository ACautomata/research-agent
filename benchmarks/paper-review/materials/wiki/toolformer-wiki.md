## 1. 研究背景

大语言模型（LM）在零样本和少样本自然语言处理任务上展现出令人印象深刻的能力，并表现出多种涌现能力。然而，这些模型存在几个固有局限性，即使进一步扩大规模也难以完全解决：(1) 无法获取关于近期事件的最新信息；(2) 倾向于产生事实幻觉；(3) 理解低资源语言存在困难；(4) 缺乏进行精确计算的数学能力；(5) 对时间流逝缺乏感知。虽然给 LM 使用外部工具（如搜索引擎、计算器、日历）是一种直观的解决方法，但现有方法要么依赖大量人工标注，要么将工具使用限制在特定任务的设置中，阻碍了工具在 LM 中的广泛应用。

## 2. 任务定义

Toolformer 的任务是训练一个语言模型，使其能够在自监督的方式下学习何时调用外部工具 API、调用哪个 API、传递什么参数，以及如何将 API 返回的结果整合到未来的 token 预测中。该方法不对下游任务做特定假设，而是在通用的语言建模数据集上学习工具使用能力，使得训练后的模型能在各种零样本下游任务中自主决定是否以及如何使用工具。

## 3. 论文要解决的核心问题

核心问题是：如何让语言模型以自监督的方式学习使用外部工具，同时满足以下两个关键需求：(1) 工具使用应通过自监督方式学习，不需要大量人工标注——这不仅是因为标注成本高昂，还因为人类认为有用的东西可能与模型认为有用的东西不同；(2) LM 不应丧失其通用性，应能自主决定何时以及如何使用哪种工具，而不是将工具使用绑定到特定任务上。

## 4. 方法总览

Toolformer 的核心流程分为三步：(1) 利用 LM 的上下文学习能力，给定每个 API 仅需少量人工编写的示例，让 LM 在一个大型语言建模数据集上自动批量生成候选 API 调用；(2) 执行这些 API 调用获取结果； (3) 使用自监督损失函数判断哪些 API 调用能真正帮助模型预测未来 token，以此作为过滤标准。只保留能降低损失（即提升预测能力）的 API 调用，然后使用这些带有 API 调用的增强数据集对 LM 本身进行微调。推理时，模型正常自回归解码直到生成 "\->" 标记，此时解码被中断，调用对应的外部 API 获取结果，插入结果和 "</API>" 标记后继续解码。

## 5. 方法关键模块

**(a) API 调用采样模块**：对于每个工具，设计一个提示模板（prompt）P(x)，包含少量人工编写的 API 使用示例，引导 LM 对输入文本 x 标注 API 调用。通过计算每个位置 i 上 <API> 特殊 token 的生成概率 p_i，保留概率超过阈值 tau_s 的位置（最多 k 个），并在每个位置从模型采样最多 m 个 API 调用。

**(b) API 调用执行模块**：执行所有生成的 API 调用，获取对应的文本结果。执行方式完全取决于 API 本身——可以是调用另一个神经网络（如 QA 系统）、执行 Python 脚本（如计算器）、或使用检索系统（如 Wikipedia 搜索）。

**(c) API 调用过滤模块**：核心过滤机制是基于损失减少的判断。计算两种加权交叉熵损失：L+_i（提供 API 调用及其结果作为前缀时的损失）和 L-_i = min(L_i(epsilon), L_i(e(c_i, epsilon)))（不调用 API 或调用但不提供结果时的损失最小值）。仅保留满足 L-_i - L+_i >= tau_f 的 API 调用，即添加 API 调用及其结果至少使损失降低 tau_f。

**(d) 模型微调模块**：将过滤后的 API 调用与原始文本交错合并，形成增强数据集 C*。使用标准语言建模目标在此数据集上微调 M。

**(e) 推理解码模块**：使用标准贪心解码，但做一修改——当 <API> token 是前 k 个最可能 token 之一时就触发 API 调用（而不只是当它是唯一最可能 token 时）。每输入最多进行一次 API 调用，以防陷入循环调用。

## 6. 关键公式与机制说明

**(1) API 调用的线性化表示**：
- e(c) = <API> a_c(i_c) </API>（不包含结果）
- e(c, r) = <API> a_c(i_c) -> r </API>（包含结果）
其中 a_c 是 API 名称，i_c 是输入，r 是结果。

**(2) 采样概率**：
p_i = p_M(<API> | P(x), x_{1:i-1})
计算 LM 在位置 i 开始 API 调用的概率。

**(3) 过滤损失函数**：
L_i(z) = -sum_{j=i}^{n} w_{j-i} * log p_M(x_j | z, x_{1:j-1})
其中 w_t = max(0, 1 - 0.2*t) 并归一化，使靠近 API 调用位置的 token 获得更高权重。

L+_i = L_i(e(c_i, r_i))  — 带上 API 调用和结果的损失
L-_i = min(L_i(epsilon), L_i(e(c_i, epsilon)))  — 基线损失的最小值
过滤条件：L-_i - L+_i >= tau_f

**(4) 推理时 API 触发策略**：当 <API> token 的生成概率排在前 k 位时触发（默认 k=10），而不是仅在它是最可能的 token 时（k=1）。

## 7. 训练与推理流程

**训练流程**：
1. 从 CCNet 数据集选取文本子集 C。
2. 对每个工具（QA、Wikipedia Search、Calculator、Calendar、Machine Translation），设计包含少量示例的 prompt。
3. 使用 GPT-J 对 C 中每个文本采样候选 API 调用位置和调用内容。
4. 执行所有候选 API 调用，获取文本结果。
5. 使用加权交叉熵损失过滤：仅保留使 L- - L+ >= tau_f 的调用。
6. 将过滤后的 API 调用与原始文本合并产生增强数据集 C*。
7. 在 C* 上使用标准语言建模目标微调 GPT-J，批次大小 128，学习率 1e-5，线性 warmup 前 10%，最多 2000 步。

**推理流程**：
1. 执行标准自回归贪心解码。
2. 当 <API> token 属于 top-k 最可能 token 之一时（k=10），触发 API 调用。
3. 模型继续生成直到生成 "\->" token。
4. 中断解码，调用对应外部 API 获取文本结果。
5. 插入结果和 "</API>" token，继续解码。
6. 每输入最多进行一次 API 调用。

## 8. 实验设置

**基础模型**：GPT-J（6.7B 参数）
**训练数据**：CCNet 子集（Wenzek et al., 2020）
**训练配置**：批次大小 128，学习率 1e-5，线性 warmup（前 10%），最大序列长度 1024，DeepSpeed ZeRO-3，8 块 NVIDIA A100 40GB，BF16 精度，最多 2000 步（每 500 步评估 PPL 选最优 checkpoint）
**外部工具**：(1) QA：Atlas（在 Natural Questions 上微调，训练用 Atlas-large，推理用 Atlas-xxl）；(2) Wikipedia Search：BM25 检索器（KILT Wikipedia dump）；(3) Calculator：Python 脚本（四则运算，结果保留两位小数）；(4) Calendar：返回当前日期；(5) Machine Translation：NLLB 600M 参数（目标语言固定为英语）+ fastText 语言检测
**默认数据生成参数**：tau_s=0.05, tau_f=1.0, k=5, m=5；Calculator/MT 例外：tau_s=0.0, tau_f=0.5, k=20, m=10；每 API 最多 25,000 训练样本
**解码配置**：贪心解码，k=10 的 API 触发策略，最多次 API 调用次数为 1
**基线模型**：GPT-J、GPT-J + CC（在 CCNet 上微调无 API）、Toolformer、Toolformer (disabled)（解码时禁用 API）、OPT (66B)、GPT-3 (175B)
**下游评估数据集**：LAMA（SQuAD/Google-RE/T-REx）、ASDiv、SVAMP、MAWPS、WebQS、Natural Questions、TriviaQA、MLQA（6 种语言）、TEMPLAMA、DATESET（自定义模板，9,400 条）
**语言建模评估**：WikiText 和 CCNet 验证子集（10,000 条随机文档）

## 9. 主要实验结果

**(1) LAMA 事实知识补全**：Toolformer 在 SQuAD (33.8)、Google-RE (11.5)、T-REx (53.5) 上全面超越同规模 GPT-J+CC 基线（19.2/5.6/33.2），也超越 GPT-3 (175B)（26.8/7.0/39.8），98.1% 例子调用 QA 工具。

**(2) 数学推理**：Toolformer 在 ASDiv (40.4)、SVAMP (29.4)、MAWPS (44.0) 上大幅超越 GPT-3 (175B)（14.0/10.0/19.8），97.9% 例子调用 Calculator；即使禁用 API（Toolformer disabled），性能（14.8/6.3/15.0）仍优于 GPT-J+CC（9.6/5.0/9.3）。

**(3) 开放域问答**：Toolformer（26.3/17.7/48.8）超越同规模基线但落后 GPT-3（29.0/22.6/65.9），99.3% 调用 Wikipedia Search；在 TriviaQA 上落后最大（差距 17.1），归因为 BM25 搜索质量不足和无法交互式优化查询。

**(4) 多语言问答 MLQA**：Toolformer 在 6 种语言上全部优于同规模基线，但未一致超越原始 GPT-J（如德语 13.5 vs 16.5），CCNet 微调导致的多语言能力退化抵消了 MT 工具收益；MT 工具使用率 63.8%-94.9%，Hindi 仅 7.3%。

**(5) 时序数据集**：TEMPLAMA 上 Toolformer (16.3) 改善有限，calendar 使用率仅 0.2%，最优策略（先查日历再查 QA）受限于单步 API 限制；DATESET 上大幅领先（27.3 vs 基线 5.9），calendar 使用率 54.8%。

**(6) 语言建模**：Toolformer (disabled) 在 WikiText (10.3) 和 CCNet (10.5) 上的 perplexity 与 GPT-J+CC 完全相同，证明 API 标注不损害语言建模能力。

**(7) 缩放规律**：工具使用能力在约 775M 参数时涌现——124M 和 355M 模型无法利用工具，775M 开始出现差距，6.7B 差距最大；Wikipedia Search 是最容易使用的工具。

**(8) 解码策略**：k=1（贪心解码）时模型展现一定校准性——选择不调用 API 的例子性能更高；k=10 时校准性丧失但整体性能提升；WebQS 上 k=3 和 k=10 性能相同。

## 10. 论文贡献总结

**(1)** 提出一种自监督方法让语言模型学会使用多种外部工具，每个工具仅需少量（handful of）人工示例，无需大量人工程序。

**(2)** 设计了三阶段流程（采样-执行-过滤），使模型能自主决定何时、调用哪个 API、传递什么参数、以及如何整合结果——工具使用不绑定到特定任务。

**(3)** 在 5 类异构下游任务上验证了方法有效性，6.7B 的 Toolformer 在多个基准上超越 175B 的 GPT-3，同时保持语言建模能力不受损害。

**(4)** 揭示了工具使用能力随模型规模涌现的规律（约 775M 参数为临界点）。

## 11. 方法特点总结

**(1) 自监督**：完全基于语言模型自身的反馈（perplexity 减少）来筛选 API 调用，无需人工标注哪些位置需要调用工具。

**(2) 任务无关**：在通用的语言建模数据上学习，不绑定特定下游任务，保留了模型的通用性。

**(3) 即插即用**：工具输入输出均以文本表示，可无缝插入文本序列，支持各种外部工具。

**(4) 低人工成本**：每个 API 仅需少量示例 prompt（通常 3-5 个输入输出对）。

**(5) 轻量级微调**：基于预训练模型继续微调而非从零训练。

**局限性**：(1) 不支持链式工具调用（一个工具的输出作为另一个工具的输入）；(2) 不支持交互式工具使用（如搜索查询改写、多结果浏览）；(3) 对输入措辞敏感；(4) 数据采样效率极低（Calculator 处理百万级文档仅得千级有用样例）；(5) 不考量 API 调用的计算成本。

## 14. 适合后续研究时重点关注的内容

**(1) 链式工具调用**：论文明确识别为局限性但未实验——TEMPLAMA 上最优策略需要先查日历再查 QA，但方法不支持。如何让模型学习多步工具组合是最值得探索的扩展方向。

**(2) 交互式工具使用**：特别是在搜索场景下——BM25 搜索质量有限，模型无法改写查询、浏览多个结果或根据反馈调整搜索策略，这与 WebGPT（Nakano et al., 2021）的能力差距显著。

**(3) 数据效率改进**：Calculator 和 MT 工具需要处理百万级文档才能获得千级有用样例，样本效率极低。可探索迭代式方法（类似 Self-Instruct、STaR）提升采样效率。

**(4) 跨架构泛化**：论文仅测试了 GPT 系列模型（GPT-2 和 GPT-J），未在 T5、LLaMA、OPT 等其他架构上验证。Toolformer 是否是通用范式仍需验证。

**(5) 工具使用能力的涌现机制**：约 775M 参数处涌现，但涌现阈值是否架构依赖、训练数据依赖？深层原因值得探究。

**(6) 校准性与高使用率的权衡**：k=1 时模型在一定程度上知道自己何时需要外部知识（校准性），但 k=10 时丧失。能否在保持高 API 使用率的同时保留或增强校准性？

**(7) 更丰富的工具集**：论文仅涵盖 5 种工具，可扩展到代码执行、数据库查询、图像生成等多模态工具。

## 15. 一句话总结

Toolformer 提出了一种自监督方法，让语言模型通过采样-执行-过滤三阶段流程在通用语料上学习自主调用外部工具 API，以 6.7B 参数在多个零样本任务上超越 175B GPT-3，同时揭示了工具使用能力在约 775M 参数处涌现的缩放规律。
